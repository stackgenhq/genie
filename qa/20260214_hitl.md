# Human-in-the-Loop (HITL) — Acceptance Criteria

> Tests for the approval store, tool wrapper approval gate, /approve endpoint, and chat UI approval/rejection flows.

---

## 17 — HITL: /approve Endpoint

### Why
The `/approve` endpoint is the HTTP API that the chat UI calls to submit approval decisions. It must validate inputs and handle edge cases.

### Problem
An unvalidated endpoint could accept malformed requests or crash on missing approvals.

### Benefit
Provides a robust HTTP interface for approval resolution with proper error codes and validation.

### Arrange
- Server running with HITL enabled (approval store configured)

### Act
```bash
curl -X POST http://localhost:8080/approve \
  -H 'Content-Type: application/json' \
  -d '{"approvalId":"nonexistent","decision":"approved"}'
```

### Assert
- Returns HTTP 500 with error message about approval not found
- Missing fields return HTTP 400
- Invalid decision values return HTTP 400

## 18 — HITL: Chat UI Approval Flow (Approve)

### Why
This is the happy-path end-to-end test for HITL. Users must be able to approve tool calls through the chat UI.

### Problem
If the approval flow is broken, users cannot authorise any non-readonly tool calls, making Genie unable to perform write operations.

### Benefit
Validates the full approval loop: tool blocked → card shown → user approves → tool executes → agent continues.

### Arrange
- Server running with HITL enabled
- Open `docs/gh-pages/chat.html` in a browser, connect

### Act
1. Send: `Create a file called test.txt with the content "hello world"`
2. An approval card should appear showing tool name and arguments
3. Click **Approve**

### Assert
- An amber/warning approval card appears with tool name (e.g., `write_file`) and arguments preview
- After clicking Approve, the card updates to show ✅ Approved
- The tool executes and the agent continues its response
- No error messages appear

---

## 19 — HITL: Chat UI Rejection Flow (Reject)

### Why
Users must be able to reject tool calls they don't want to execute. The agent must handle rejections gracefully.

### Problem
Without rejection support, users would have no way to prevent unwanted operations short of killing the server.

### Benefit
Gives users full control over agent actions and validates the agent's ability to recover from rejected tool calls.

### Arrange
- Server running with HITL enabled
- Open `docs/gh-pages/chat.html` in a browser, connect

### Act
1. Send: `Run the command "echo hello" in the terminal`
2. An approval card should appear showing tool name and arguments
3. Click **Reject**

### Assert
- An amber/warning approval card appears with the tool name (e.g., `execute_code`)
- After clicking Reject, the card updates to show ❌ Rejected
- The agent reports the rejection gracefully (e.g., "The tool call was rejected")
- No error bubbles or crashes occur
