# Chat UI — Acceptance Criteria

> Blackbox acceptance tests for the Genie chat page ([chat.html](https://stackgenhq.github.io/genie/chat.html)). Covers connect, send, microphone, read aloud, fullscreen, keyboard shortcuts, and vibration.

---

## 1 — Connect to Server

### Why
The chat UI is the primary way users talk to Genie. Connecting validates the AG-UI handshake and CORS.

### Problem
Without a working connection, users cannot send messages or use any chat features.

### Benefit
Confirms the browser can reach the running Genie server before using other features.

### Arrange
- Genie server running (e.g. `./build/genie grant` on port 8080)
- Open [chat.html](https://stackgenhq.github.io/genie/chat.html) (or local `docs/chat.html`)

### Act
1. Set the endpoint to `http://localhost:8080` (or your server URL).
2. Click **Connect**.

### Assert
- Connection badge shows **Connected** (green).
- A system message appears (e.g. "Genie is ready at …").
- The main input and **Send** button are enabled; you can type and send a message.

---

## 2 — Send Message and Receive Reply

### Why
Core flow: user sends a message and the assistant streams a reply.

### Problem
Broken send or stream handling would make the chat unusable.

### Benefit
Validates the full request/response and streaming pipeline.

### Arrange
- Connected to the server (Test 1).

### Act
1. Type a short message (e.g. "Hello, what can you do?").
2. Press **Enter** or click the send (arrow) button.

### Assert
- Your message appears as a user bubble on the right.
- A reply appears as an assistant bubble on the left (streaming or final).
- No error bubbles; input is cleared and ready for the next message.

---

## 3 — Microphone (Voice Input)

### Why
Users can speak instead of typing for accessibility and convenience.

### Problem
Without voice input, keyboard-only or speech-first users are disadvantaged.

### Benefit
Enables hands-free or faster input when the browser supports speech recognition.

### Arrange
- Connected to the server (Test 1).
- Use a browser that supports Web Speech Recognition (e.g. Chrome, Edge).
- Allow microphone when prompted.

### Act
1. Click the **microphone** button (next to the send button).
2. Speak a short phrase (e.g. "What is the weather?").
3. Click the microphone again to stop, or leave it to stop automatically.

### Assert
- The button shows a recording state (e.g. red/pulsing) while listening.
- Spoken text appears in the input field (possibly with "[listening…]").
- After stopping, the text remains in the input; you can edit and send with Enter.

---

## 4 — Read Aloud (Speech Synthesis)

### Why
Users can hear assistant replies read aloud for accessibility or multitasking.

### Problem
Long or dense replies are harder to consume for some users without TTS.

### Benefit
Makes replies accessible and allows eyes-free listening.

### Arrange
- Connected and at least one assistant reply visible in the chat (Test 2).

### Act
1. Hover over an assistant message bubble.
2. Click the **speaker** (read aloud) button on that bubble.

### Assert
- The reply is read aloud by the browser’s speech synthesis.
- The speaker button shows an active state (e.g. "Stop reading") while speaking.
- Clicking again (or pressing **Esc**) stops playback.

---

## 5 — Fullscreen

### Why
Users can expand the chat to fullscreen for focus or small screens.

### Problem
Cluttered or small viewports make long conversations harder to follow.

### Benefit
Maximizes chat area and reduces distractions.

### Arrange
- Open chat.html in a browser that supports the Fullscreen API.

### Act
1. Click **⛶ Fullscreen** in the header.

### Assert
- The chat container goes fullscreen (no browser chrome).
- The button label changes to **Exit fullscreen** (or similar).
- Clicking it again (or pressing **Esc**) exits fullscreen.

---

## 6 — Keyboard Shortcuts

### Why
Power users can send, focus, and stop without using the mouse.

### Problem
Mouse-only usage is slower for frequent actions.

### Benefit
Faster send, focus, and stop with consistent shortcuts.

### Arrange
- Connected to the server (Test 1); input focused or focusable.

### Act & Assert
- **Enter** (without Shift): sends the current message. Assert: message is sent.
- **Shift+Enter**: inserts a new line. Assert: cursor moves to next line, message not sent.
- **⌘K** (Mac) or **Ctrl+K** (Windows/Linux): focuses the main chat input. Assert: focus is in the input.
- **Esc**: stops streaming if a reply is in progress, or stops read-aloud if playing. Assert: streaming or TTS stops.

---

## 7 — Vibration Toggle (Mobile)

### Why
On supported devices, vibration can be turned off for quiet environments.

### Problem
Unexpected vibration can be disruptive.

### Benefit
Users control whether alerts trigger device vibration.

### Arrange
- Open chat.html on a device or browser that supports `navigator.vibrate`.
- Optional: ensure at least one approval or clarification would trigger vibration (or use the in-app toggle to test state).

### Act
1. Find the **Vibrate on alerts** control (e.g. in the hint area below the input or in a small settings link).
2. Turn vibration **off** (toggle or link).
3. Trigger an event that would normally vibrate (e.g. an approval request), or reload and confirm the setting persisted.

### Assert
- When vibration is off, the device does not vibrate on approval/clarification/reply notifications.
- When vibration is on (default), a short vibration occurs on those events (if supported).

---

## 8 — Install as App (PWA)

### Why
Users can install the chat as a standalone app from the browser.

### Problem
Without install, the chat is always opened inside a browser tab.

### Benefit
App-like icon and window; can open directly to chat.

### Arrange
- Open [chat.html](https://stackgenhq.github.io/genie/chat.html) in a browser that supports PWA install (e.g. Chrome, Edge).

### Act
1. Use the browser’s **Install** or **Add to Home Screen** option (e.g. in the address bar or menu).
2. Complete the install.
3. Open the installed app.

### Assert
- The app opens to the chat page (e.g. /genie/chat.html or /chat.html).
- The app runs in a standalone window (no browser UI).
- Chat connect/send flow works the same as in the browser tab.
